# 适配新设备型号

本指南面向想把本插件扩展到其他松下设备型号的开发者。所需技能：Python 基础 + 能用命令行。

---

## 第一步：确认你的设备信息

运行 `tools/dump_device_params.py`，输出的每台设备包含：

| 字段 | 含义 |
|------|------|
| `deviceId` | 格式 `MAC_CATEGORY_SUFFIX`，category 决定设备大类 |
| `devSubTypeId` | 决定走哪个 API 端点，如 `DCERV-03`、`MIDERV`、`NEWDCERV-01` |
| `deviceMNO` | 设备型号，如 `FY-35ZJD2C` |
| `statusAll` | 上次 GET 缓存的完整状态字段 |

记下你的 `devSubTypeId`，这是后续所有操作的基准。

---

## 第二步：找到正确的 GET 端点

运行 `tools/probe_endpoints.py` 自动探测所有已知端点的响应：

```bash
python3 tools/probe_endpoints.py
```

脚本会对账号下每台设备依次测试以下端点，并打印结果：

| 端点 | 已知适用设备 |
|------|------------|
| `ADevGetStatusDCERV` | DCERV-03（本项目已实测） |
| `ADevGetStatusNewDCERV` | NEWDCERV-* 系列 |
| `ADevGetStatusMidERV` | MidERV 系列 |
| `ADevGetStatusSmallERV` | SmallERV / 小型新风 |
| `ADevGetStatusNeedsAP` | NeedsAP 系列 |
| `ADevGetStatusJDNeedsAP` | JD NeedsAP 系列 |
| `ADevGetStatusInfoERV` | InfoERV 系列 |
| `ADevGetStatusInfoFloorPlacedERV` | 落地式 ERV |
| `ACDevGetStatusInfoAW` | AW 系列（空调新风一体机） |

**成功标志**：某一行打印 ✅，响应里有 `results` 字典且包含 `runSta` 等控制字段。

**如果全部 ❌**：你的设备对应端点尚未收录。可用以下方法推断：
1. 记下第一步的 `devSubTypeId`（如 `FOOBAZ-01`），按命名规律猜端点：`ADevGetStatusFoobaz`
2. 在旧版 App APK（见第三步）里搜索：`grep -r "devSubTypeId" app_src/sources/ -l`，找到对应类后再搜 `URL_ADEV_GET_STATUS`
3. 找到新端点后，手动加到 `tools/probe_endpoints.py` 的 `GET_ENDPOINTS` 列表末尾再重跑

---

## 第三步：获取 SET payload 字段表

有了可用的 GET 端点后，GET 返回的 `results` 就是该设备的完整状态。区分两类字段：

**只读传感器**（不能写入 SET）：通常是物理传感器读数，名称规律如：
- `*PMC`（PM2.5）、`*TeC`（温度）、`*HumC`（湿度）、`*CO2*`（CO2）
- `*TL`（滤网剩余寿命）、`offline`、`serverTime`、`hVer`、`sVer`

**可写控制字段**：`runSta`、`runM`、`airVo`、`holM`、`preM` 等，以及对应设备子类的特有字段。

如果能拿到无 Payegis 保护的旧版 App APK（通常是 3.x 版本），可用 jadx 反编译直接查看源码中的 Java bean：

```bash
jadx -d app_src/ your_old_app.apk
# 搜索你的 devSubTypeId 对应的 bean 类
grep -r "devSubTypeId" app_src/sources/ -l
grep -r "generateSetStatusJson\|createADevSetStatus" app_src/sources/ -l
```

关键类通常是 `*DevStatusSetBean.java` 和对应的 `*ParamSettingUtil.java`，里面列出了所有可写字段及默认值（通常为 255 = "不改"）。

---

## 第四步：验证 SET 端点

**SET 端点命名规律**：GET 端点把 `Get` 换成 `Set` 即可，例如：
- `ADevGetStatusNewDCERV` → `ADevSetStatusNewDCERV`
- `ADevGetStatusMidERV` → `ADevSetStatusMidERV`

在运行 `tools/test_set_field.py` 之前，先修改脚本顶部的三处配置（搜索"用户配置区"注释）：

```python
# 1. 改成你的端点（从上一步的 probe 结果复制）
GET_ENDPOINT = "ADevGetStatusNewDCERV"   # ← 改这里
SET_ENDPOINT = "ADevSetStatusNewDCERV"   # ← 改这里（Get→Set）

# 2. 粘贴你的 deviceId（留空则自动用账号第一台设备）
DEVICE_ID = "AABBCC112233_0800_FOOBAR-01"  # ← 改这里，或留 ""

# 3. 替换字段列表（来自上一步的 App 源码 *DevStatusSetBean.java）
PAYLOAD_DEFAULTS: dict = {
    "runSta": 255,
    "runM": 255,
    # ... 你设备的所有可写字段，默认 255
    # 定时器小时/分钟例外，用 127
}
```

配置好后运行：

```bash
# 先确认 GET 正常，设备在线
python3 tools/test_set_field.py get

# 发全 255 payload 验证 SET 端点可达（不改任何实际字段）
python3 tools/test_set_field.py set

# 测试开关机（最安全的实际控制测试）
python3 tools/test_set_field.py set runSta 0   # 关机
python3 tools/test_set_field.py set runSta 1   # 开机
```

**payload 构造原则**：
1. 所有可写字段全部包含，未要修改的用 255（skip 值）填充
2. 定时器字段的小时/分钟用 127 填充（`tH1-6`、`tMin1-6`）
3. 字段名区分大小写，从 App 源码或 GET 响应直接复制

若 SET 返回 `{"results": {"todoId": N}}` 且设备在 3 秒内响应，说明成功。若返回 todoId 但设备不执行，参考下一节排查。

---

## 排查：todoId 返回但设备不执行

这是最常见的问题，检查顺序：

1. **payload 字段不完整**：对照 App 源码的 bean 类，检查是否有遗漏的必须字段。本项目 DCERV-03 就是因为缺少 `preSet`、`pmSen`、`coSen`、`tvSen`、`userSupWind`、`userExhWind`、`aircJoi` 和定时器 `tM/tWind` 字段而不执行。

2. **端点选错**：`devSubTypeId` 是 `DCERV-03` 就用 DCERV 端点，是 `NEWDCERV-*` 就用 NewDCERV 端点，不能混用。

3. **字段值范围不对**：某些字段的合法值范围比较窄，发超出范围的值服务器可能接受但设备忽略。从 GET 响应的实际值推断范围。

---

## 第五步：实现 HA 实体

确认 GET/SET 工作后，参考现有代码接入 HA：

| 需求 | 参考文件 |
|------|---------|
| 读取传感器 | `sensor.py` |
| 开关控制 | `switch.py` + `select.py` 的 `_build_dcerv_payload()` |
| 模式/风量选择器 | `select.py` 的 `_FreshAirSelect` 基类 |
| 空调温控 | `climate.py` |

新设备子类型最小工作量：
1. 在 `sensor.py` 里增加你的字段映射
2. 在 `switch.py` 或 `select.py` 里增加一个新的 `_build_*_payload()` 函数
3. 在 `coordinator.py` 里按 `devSubTypeId` 分支选择 GET 端点
